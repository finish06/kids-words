## Technical Discoveries
- 2026-04-18: **Python 3.13 + `coverage` + `pytest-asyncio` + `httpx.AsyncClient` + FastAPI `dependency_overrides` silently under-reports coverage on async route handler bodies.** Tests pass and assert correctly (route handlers *are* running), but the coverage tracer misses the function bodies because trace context is lost across asyncio task switches during DI resolution. Concrete symptom: `results.py` reported as 40% (lines 39-96 missed — the whole body) with all 5 results tests green. Python 3.14 does not have this issue (route coverage jumps to 98-100% on the same tests). Fix: align CI Python to 3.14; production Docker image can stay on 3.13 since this is a test-harness-only pathology. Source: cycle-11 (PR #16).
- 2026-04-18: **mypy flow analysis on reused SQLAlchemy result variables is fragile.** Reusing the name `result` across multiple `session.execute(select(X))` calls in the same function — where X differs each time (Pattern, then BaseWord, then WordCombo) — causes mypy to narrow `existing = result.scalar_one_or_none()` to the *first* type and then flag attribute errors on the later loops. The code runs fine (SQLAlchemy returns the correct type), but the type checker can't follow. Fix: use distinct variable names per query (`pat_result`, `bw_result`, `combo_result`) and add explicit type annotations on the `existing_*` locals (e.g., `existing_combo: WordCombo | None = combo_result.scalar_one_or_none()`). Source: cycle-12, `app/seed_word_builder.py`.
- 2026-04-04: OpenMoji CDN correct path is `cdn.jsdelivr.net/gh/hfg-gmuend/openmoji/color/618x618/{code}.png` — the npm path (`cdn.jsdelivr.net/npm/openmoji@15.1/`) returns 404. Source: trial and error with curl.
- 2026-04-04: SQLAlchemy `Word.results` relationship needs explicit `foreign_keys` when MatchResult has two FKs to words table. Source: mapper configuration error on first test run.
- 2026-04-04: Playwright E2E tests must live outside the frontend package (project root `tests/e2e/`) — Vitest's `globals: true` contaminates the Playwright runner with `@vitest/expect` Symbol conflicts. Source: runtime error on first Playwright run.
- 2026-04-04: Hatchling needs `[tool.hatch.build.targets.wheel] packages = ["app"]` when the package name doesn't match the directory name. Source: Docker build failure.

- 2026-04-04: Chose `Annotated[AsyncSession, Depends(get_db)]` pattern over bare `Depends()` in route params to satisfy ruff B008 rule.

## What Worked
- Away mode: completed Phase 0 + Phase 1 + Phase 2 (24/32 tasks) in a single session
- OpenMoji CDN for images: zero local storage, instant availability, kid-friendly style
- Vitest + React Testing Library for frontend unit tests: fast, integrates with Vite config
- Docker Compose with volume mounts for hot reload during development

## What Didn't Work
- `npm ci` in Docker fails when local Node version (v25) differs from container (v22) — lock file incompatibility. Switched to `npm install` in Dockerfile.
- Port 5432 already in use on dev machine — had to remap postgres to 5433.
- Vite dev server in Docker needs explicit `host: '0.0.0.0'` in config + `usePolling: true` for file watching.

- 2026-04-18 [cycle-13 PAT finding]: **English morphology is generative — any pattern-matching game built on verb/adjective/noun suffixes must handle multiple-valid-answers explicitly.** User caught: "Paint (un-, -ed, -s). Both -ed and -s work. Paints or painted. Both words." PAINT accepts -ED/-S/-ER/-ING/RE- in real English; our single-correct-pattern challenge model treats all but "correct" as wrong → false-negative feedback violates non-punitive UX promise. Four honest fixes: (A) multi-correct answers, (B) curate to single-pattern-only bases [infeasible — too few candidates], (C) distractor filter [incomplete], (D) clue-based challenges ["a person who paints" → -ER, unambiguous]. User chose D combined with M8 Phonetics (tap-to-hear for pre-readers). Worth promoting to ~/.claude/add/library.md at next retro: "word-morphology games require disambiguation via context (clue, target word, or instruction) — not tile-matching alone."
